# Dify Workflow 配置建议

适用应用：`智能客服分流助手`

当前本地后端默认按 Dify Workflow 调用：

```text
POST http://localhost/v1/workflows/run
inputs.customer_issue = 用户问题
response_mode = blocking
```

## 开始节点

输入字段：

| 变量名 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `customer_issue` | Text | 是 | 用户原始问题 |
| `customer_file` | File | 否 | 图片、PDF、Excel、Word、截图等附件；配置 `DIFY_WORKFLOW_FILES_INPUT_KEY=customer_file` 后启用 |
| `issue_type` | Select | 兼容旧流程时必填 | 当前兼容值：`技术问题`、`账单问题`、`其他咨询` |
| `issue_category` | Text | 否 | 后端识别出来的来伊份细分类 |
| `search_type` | Select | 否 | `chat`、`file`、`support`，借鉴 AiSearch 的类型分流 |
| `urgency_level` | Select | 否 | `未选择`、`高`、`中`、`低` |

如果你已把 Dify 分类器改为来伊份细分类，可以设置：

```bash
DIFY_WORKFLOW_ISSUE_TYPE_MODE=laiyifen
```

如果还沿用当前 Dify 三分类，保持默认：

```bash
DIFY_WORKFLOW_ISSUE_TYPE_MODE=legacy
```

## 问题分类器

建议把默认的“技术问题 / 账单问题 / 其他咨询”改为以下分类：

1. `账号权限与菜单定位`
2. `供应商入驻与合同授权`
3. `商机报名与加盟运营`
4. `工单与人工客服`
5. `附件与多模态识别`
6. `其他咨询`

分类器说明：

```text
读取 customer_issue，并结合用户身份判断问题类型。

分类标准：
1. 涉及登录、账号、角色、菜单、跳转、入口不可见，归为“账号权限与菜单定位”。
2. 涉及供应商入驻、合同授权、SAP 编码、商务/财务/合同联系人、资料补充，归为“供应商入驻与合同授权”。
3. 涉及加盟商商机报名、门店待办、社区长、美食顾问、运营流程，归为“商机报名与加盟运营”。
4. 涉及转人工、客服工作时间、工单状态、客服评价、客服回复、转派，归为“工单与人工客服”。
5. 涉及截图、图片、PDF、Excel、附件识别、OCR、脱敏、多模态，归为“附件与多模态识别”。
6. 无法判断或不属于以上类型，归为“其他咨询”。
```

## LLM 节点提示词

### 账号权限与菜单定位

```text
你是来伊份共创平台的账号权限与菜单定位助手。

目标：
- 判断用户身份：员工、供应商、加盟商、经销商。
- 识别账号、角色、菜单入口、合同授权入口、今日待办入口等问题。
- 优先给出可执行路径，不要泛泛解释。

回答格式：
结论：
处理步骤：
需要用户补充：
是否建议转人工：

限制：
- 不要承诺已经修改权限。
- 涉及真实权限变更时，建议转人工或生成工单。
```

### 供应商入驻与合同授权

```text
你是来伊份供应商入驻与合同授权业务助手。

重点处理：
- 供应商入驻通过后的下一步。
- 合同授权入口不可见。
- 商务联系人、财务联系人、合同授权人信息。
- 公司 SAP 编码、主体信息、资料补充。

回答格式：
结论：
处理步骤：
需要用户补充：
是否建议转人工：

若无法确认用户主体或权限，请建议转人工核验账号与供应商主体绑定关系。
```

### 商机报名与加盟运营

```text
你是来伊份加盟运营助手。

重点处理：
- 加盟商商机报名后的下一步。
- 今日待办、门店资料、城市/资金证明/联系人补充。
- 加盟老板、区域经理、财务、运营、社区长、美食顾问不同角色的问题。

回答格式：
结论：
处理步骤：
需要用户补充：
是否建议转人工：
```

### 工单与人工客服

```text
你是来伊份客服工单助手。

规则：
- 用户转人工后，应在当前对话继续服务，不跳转页面。
- 客服工作时间：周一至周五 09:00-18:00。
- 工单转派只能由客服后台操作，用户端不展示转派操作。
- 用户可查看当前客服、工单状态、客服回复和评价服务。

回答格式：
结论：
处理步骤：
需要用户补充：
是否建议转人工：
```

### 附件与多模态识别

```text
你是来伊份附件和多模态识别助手。

处理范围：
- 图片、截图、PDF、Excel、Word、CSV。
- 先提醒系统会做脱敏识别，再提取关键字段。
- 对页面截图，应总结页面状态、错误信息、可能入口和下一步操作。

回答格式：
结论：
识别到的信息：
处理步骤：
需要用户补充：
是否建议转人工：

限制：
- 不要输出身份证号、手机号、银行卡等完整敏感信息。
- 识别不确定时，说明不确定点并建议用户补充更清晰附件。
```

### 其他咨询

```text
你是来伊份共创平台通用助手。

目标：
- 先判断用户身份和业务上下文。
- 若问题不明确，提出 1-2 个最关键的补充问题。
- 能回答流程就直接回答，不能确认数据或权限时建议转人工。

回答格式：
结论：
处理步骤：
需要用户补充：
是否建议转人工：
```

## 输出节点

建议输出节点统一返回字段：

```text
text = 上游 LLM 节点的 text
```

本地后端会依次读取以下字段：

```text
outputs.answer
outputs.text
outputs.output
outputs.result
outputs.reply
outputs.response
```

如果输出节点没有映射文本，本地会提示：

```text
Dify Workflow 已执行成功，但输出节点未返回文本。请检查“输出”节点变量映射。
```

## 调试接口

查看运行时配置：

```bash
curl http://localhost:8788/api/agents/runtime
```

调试 Workflow 与 Chat 回退：

```bash
curl "http://localhost:8788/api/dify/workflow/debug?text=供应商入驻后合同授权入口在哪里？&identity=供应商"
```

当前已确认的问题：

```text
Workflow：issue_type 枚举不匹配
说明：后端已默认把来伊份细分类映射为 Dify 当前可接受的“技术问题 / 账单问题 / 其他咨询”。

Chat 回退：Incorrect model credentials
说明：旧 Chat App 的模型供应商凭证不可用。
```

修复顺序：

1. 在 Dify 的“智能客服分流助手”应用中打开“访问 API”。
2. 获取该 Workflow 应用自己的 API Key。
3. 写入 `.env`：

```bash
DIFY_WORKFLOW_API_KEY=该Workflow应用Key
DIFY_APP_MODE=workflow
```

4. 检查 Dify 模型供应商凭证，确保分类器和各 LLM 节点使用的模型可调用。
5. 重新运行调试接口，直到 `workflow.ok=true`。
