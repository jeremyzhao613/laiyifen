原始测试结果：`output/playwright/question-comparison-results.json`

## 测试方法

用 Playwright 分别打开线上和本地页面，对同一批问题逐条发送，记录：

- 页面返回的回答文本
- 新增可点击动作按钮
- XHR / Fetch 接口调用状态
- 每个问题的截图

判定口径：

- `已命中`：页面有业务化回答，或能进入数据/工单/导航处理链路。
- `弱命中`：能回答一部分，但缺少下一步动作，或回答停留在基础客服话术。
- `未命中`：返回“没有找到相关问题的答案”等兜底内容。
- `前端拦截`：没有发送到后端，而是前端提示用户补充必要输入。

## 总体结论

| 对象 | 已命中 | 弱命中 | 未命中 | 前端拦截/无后端消息 |
| --- | ---: | ---: | ---: | ---: |
| 线上现网 | 1 | 3 | 12 | 0 |
| 本地新版 | 15 | 0 | 0 | 1 |

结论：

1. 本地新版在共创平台业务问题上的覆盖明显更完整，尤其是结算入口、合同授权、今日待办、资料补传、SAP 主体、加盟商机、门店审核、经销订单、库存预警等问题。
2. 本地新版不仅回答问题，还能返回业务动作按钮，例如“打开结算页面”“打开合同授权页面”“打开供应商资料页”“生成客服工单”。
3. 线上现网对原有 FAQ 能回答一部分，例如供应商入驻资料，但对新增业务场景大多走“没有找到答案”的兜底。
4. 本地新版仍有需要优化的点：账号变更、加盟资料提交类问题容易被安全/任务执行流程拦截；图片附件链路已通，但真实视觉模型内容没有完全接入。

## 关键样例

### 结算入口问题

问题：`这个账号为什么看不到结算入口？`

线上结果：

- 返回“非常抱歉，没有找到相关问题的答案”。
- 只提供“复制 / 有帮助 / 没帮助 / 转人工客服 / 评价服务”。

本地结果：

- 给出账号身份、权限、合同状态、流程进度等排查步骤。
- 额外提供“打开结算页面”和“生成客服工单”。
- 已验证“打开结算页面”会跳转到：
  `https://scm.test.laiyifen.com/webadmin_vue/collaborationPlatform.html#/finance/settlement`

### 合同授权问题

问题：`合同授权卡在待确认，下一步怎么做？`

线上结果：

- 未命中，返回无答案兜底。

本地结果：

- 能说明联系人信息、权限、入驻流程、待确认节点等排查方向。
- 提供“打开合同授权页面”和“生成客服工单”。

### 人工客服问题

问题：`我要转人工处理投诉，怎么接入客服？`

线上结果：

- 能显示“正在为您转接人工客服”，属于基础命中。

本地结果：

- 生成工单 `TK-20260630-004`。
- 显示当前处理人、飞书通知状态。
- 提供“查看工单”动作。

### 图片附件问题

问题：`我上传了页面截图，能帮我识别错误提示吗？`

本地无附件时：

- 前端拦截并提示先上传图片、截图或相关文件。

本地上传截图后：

- 页面显示附件 `home.png` 并支持预览。
- 回答中说明已收到附件，但模型侧还未拿到视觉内容。
- 提供“打开供应商资料页”和“生成客服工单”。

这说明本地新版已经完成了上传、预览、消息上下文和工单动作，但还需要把真实视觉模型或 Dify 文件变量完整接上。

## 本地新版更完善的能力

### 1. 业务入口导航

本地回答会根据问题生成页面动作：

| 问题类型 | 动作 |
| --- | --- |
| 结算、账单、发票 | 打开结算页面 |
| 合同、授权、签章 | 打开合同授权页面 |
| 待办、审核、待确认 | 打开今日待办 |
| 供应商资料、SAP、营业执照 | 打开供应商资料页 |
| 加盟商机、报名 | 打开商机报名页 |
| 经销订单、库存、对账 | 打开订单与对账页 |

这部分是线上现网最缺的能力。线上回答即使命中 FAQ，也没有把用户带到下一步操作。

### 2. 工单与人工客服

本地可以在回答后生成客服工单，并显示：

- 工单编号
- 当前处理人
- 飞书通知状态
- 查看工单动作

线上能显示人工客服接入话术，但没有在本轮测试里展示工单编号和飞书通知状态。

### 3. 身份化快捷问题

本地按身份提供快捷问题：

- 供应商
- 加盟商
- 员工
- 经销商

线上当前只展示少量供应商 FAQ。建议线上同步本地的问题池，提升首屏引导命中率。

### 4. 数据查询和权限提示

本地对“今日待办”“本周待处理订单”等问题会进入业务查询链路。测试账号没有可读数据时，会明确提示暂无数据，并标注数据来源。

线上对应问题大多未命中。

### 5. 附件体验

本地支持：

- 上传图片/文件
- 图片缩略图
- 图片预览
- 附件随问题一起进入上下文
- 没有附件时阻止视觉类问题盲发

需要继续完善的是视觉模型识别结果本身。

## 风险和待优化项

| 优先级 | 问题 | 建议 |
| --- | --- | --- |
| P0 | GitHub Action 当前使用 Node 20，但 `wrangler@4.105.0` 要求 Node `>=22.0.0` | 部署 workflow 改为 Node 22 |
| P1 | q01 账号登录问题被本地识别为高风险确认 | 将“忘记密码/手机号找回/账号找回”优先路由到知识问答，只有“变更绑定手机号/修改主体信息”进入风险确认 |
| P1 | q10 “提交加盟资料后下一步”被误判为任务提交 | 增加“已提交后询问进度/下一步”的知识路由规则，避免进入任务执行确认 |
| P1 | 图片上传后模型提示未拿到视觉内容 | 检查 Dify Workflow 文件变量、Chat 多模态配置，确保真实图片进入视觉节点 |
| P1 | 线上大部分业务场景未命中 | 同步本地快捷问题、业务路由、导航动作和兜底知识库 |
| P2 | q06/q13 数据查询返回暂无数据 | 上线前接真实账号权限和业务接口，区分“确实无数据”和“当前环境无权限/无模拟数据” |

## 上线建议

1. 先把本地的业务导航动作接入线上：结算、合同授权、待办、供应商资料、商机报名、订单对账。
2. 同步本地快捷问题池，按身份展示高频问题。
3. 将线上“没有找到答案”的兜底改为本地这套业务化兜底：先给排查步骤，再给页面入口或工单。
4. 优先修正 q01、q10 的路由误判，避免用户问流程说明时被要求“确认执行”。
5. 完成图片附件到 Dify 视觉节点的打通，再开放“截图识别错误提示”的正式能力。
6. 部署流水线升级 Node 22 后再合并 Cloudflare Pages 自动发布配置。
